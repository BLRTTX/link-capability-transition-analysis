# 能力覆盖分析

## 功能说明

分析新链路是否完全覆盖老链路的所有业务能力。

---

## 业务能力清单

### 输出格式

```
| 能力编号 | 业务能力描述 | 老链路实现代码位置 | 触发条件 |
|---------|------------|-----------------|---------|
```

### 示例

```
| 能力编号 | 业务能力描述 | 老链路实现代码位置 | 触发条件 |
|---------|------------|-----------------|---------|
| CAP-001 | 用户信息查询 | OrderService:45 | 创建订单前 |
| CAP-002 | 订单金额计算 | OrderService:67 | 创建订单时 |
| CAP-003 | 支付创建 | OrderService:89 | 订单创建成功后 |
| CAP-004 | 库存扣减 | OrderService:101 | 支付成功后 |
| CAP-005 | 订单状态更新 | OrderService:112 | 各阶段回调 |
```

---

## 能力覆盖检查

### 输出格式

```
| 能力编号 | 业务能力 | 老链路实现方式 | 新链路实现方式 | 是否覆盖 | 差异说明 | 风险等级 |
|---------|---------|--------------|--------------|---------|---------|---------|
```

### 检查维度

- **完全覆盖**: 新链路以相同或等价方式实现该能力
- **部分覆盖**: 新链路实现了该能力,但行为有差异
- **未覆盖**: 新链路完全没有实现该能力
- **新增能力**: 新链路新增的能力(老链路没有)

### 示例

```
| 能力编号 | 业务能力 | 老链路实现方式 | 新链路实现方式 | 是否覆盖 | 差异说明 | 风险等级 |
|---------|---------|--------------|--------------|---------|---------|---------|
| CAP-001 | 用户信息查询 | UserService.getUserInfo | NewUserService.getUserInfo | 完全覆盖 | 接口升级,行为一致 | 低 |
| CAP-002 | 订单金额计算 | 本地计算逻辑 | 本地计算逻辑 | 完全覆盖 | 计算规则一致 | 低 |
| CAP-003 | 支付创建 | PaymentService.createPayment | NewPaymentService.createPaymentV2 | 部分覆盖 | 新增交易ID返回 | 中 |
| CAP-004 | 库存扣减 | InventoryService.deductStock | - | 未覆盖 | 完全缺失 | 高 |
| CAP-005 | 订单状态更新 | 本地状态机 | 分布式状态机 | 完全覆盖 | 实现方式优化 | 低 |
```

## 边界条件分析

### 检查项

检查特殊场景的能力覆盖:

#### 1. 异常处理

```
| 异常场景 | 老链路处理方式 | 新链路处理方式 | 是否一致 | 差异说明 |
|---------|--------------|--------------|---------|---------|
```

示例维度:
- 子调用失败的处理
- 参数校验失败的处理
- 数据库操作失败的处理
- 网络超时的处理

#### 2. 超时重试

```
| 操作 | 老链路超时时间 | 老链路重试策略 | 新链路超时时间 | 新链路重试策略 | 是否一致 |
|-----|-------------|-------------|-------------|-------------|---------|
```

示例维度:
- RPC调用超时时间
- 数据库查询超时时间
- 外部API调用超时时间
- 重试次数和重试间隔

#### 3. 幂等性

```
| 操作 | 老链路幂等性 | 新链路幂等性 | 是否一致 | 差异说明 |
|-----|------------|------------|---------|---------|
```

示例维度:
- 接口是否支持幂等重复调用
- 数据库操作是否幂等
- 分布式事务的幂等保障

#### 4. 事务性

```
| 操作 | 老链路事务策略 | 新链路事务策略 | 是否一致 | 差异说明 |
|-----|--------------|--------------|---------|---------|
```

示例维度:
- 本地事务范围
- 分布式事务处理方式
- 事务失败回滚策略
- 最终一致性保障

#### 5. 并发控制

```
| 场景 | 老链路并发控制 | 新链路并发控制 | 是否一致 | 差异说明 |
|-----|--------------|--------------|---------|---------|
```

示例维度:
- 并发锁机制
- 数据库乐观锁/悲观锁
- 分布式锁的使用

#### 6. 降级熔断

```
| 服务 | 老链路降级策略 | 新链路降级策略 | 是否一致 | 差异说明 |
|-----|--------------|--------------|---------|---------|
```

示例维度:
- 熔断触发条件
- 降级后的兜底逻辑
- 服务降级开关配置

## 分析方法

### 1. 能力识别

从代码或文档中识别业务能力:

```javascript
// 老链路能力识别
{
  "老链路能力": [
    {
      "能力编号": "CAP-001",
      "能力描述": "用户信息查询",
      "实现代码": "OrderService.java:45",
      "触发条件": "创建订单前"
    },
    {
      "能力编号": "CAP-002",
      "能力描述": "订单金额计算",
      "实现代码": "OrderService.java:67",
      "触发条件": "创建订单时"
    }
  ]
}
```

### 2. 能力映射

将老链路能力映射到新链路:

- 查找新链路中对应能力的实现
- 对比实现方式的差异
- 评估能力覆盖程度

### 3. 差异分析

分析能力差异:

- 完全缺失的能力(高风险)
- 部分实现的能力(中风险)
- 新增的能力(需评估影响)
- 实现方式变化的能力(需评估影响)

### 4. 风险评估

评估能力差异的风险:

**高风险**:
- 关键业务能力缺失
- 核心流程不完整
- 数据一致性问题

**中风险**:
- 能力部分实现
- 实现方式变化
- 性能影响

**低风险**:
- 新增能力
- 实现优化
- 非核心能力变化
